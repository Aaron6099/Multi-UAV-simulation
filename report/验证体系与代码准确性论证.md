# 代码准确性验证论证（V&V 体系）

> 面向"如何完整验证 MPC 编队代码的准确性"这一问题的方法论与证据。
> **核心论点**：代码准确性不由任何单一"黄金仿真"证明，而由**分层 V&V 证据链**证明；
> 其中 Simulink 移植件的价值在于它是**独立的算法交叉验证**（不同栈/不同求解器/不同被控对象），
> PX4-Gazebo SITL 的价值在于它直接运行**部署代码本体**（最高集成保真度）。二者互补，缺一不可。

---

## 0. 方法论：为什么不是"把 PX4-Gazebo 迁移进 Simulink"

| 命题 | 说明 |
|---|---|
| 验证对象是**部署代码本体** | 真机飞的是 ROS2/`mpc_node.py` + acados + PX4 固件。验证它最高保真的方式是让**它本身**在 PX4-Gazebo SITL 中运行（同一固件二进制 + 同一节点，仅物理对象非真机）。 |
| "完整迁移"得到的是**副本** | 把 PX4-Gazebo 重写进 Simulink，得到第二份实现；用副本验证原件只能发现**差异**，且忠实迁移会使 bug 一并迁移（共因失效），逻辑上不证明准确性。 |
| Simulink 的价值=**独立性** | 移植件用不同求解器（自写 ADMM ≠ acados HPIPM）、不同建模、不同被控对象，却得到**相同行为**=交叉印证。"完整忠实迁移"会**摧毁**这一独立性，退化为 SITL 的劣化复制。 |
| "完整"=**整条证据链** | 单一仿真给不了"完整验证"。下面 §1 的分层矩阵才是可辩护的"完整性"。 |

## 1. V&V 验证矩阵（核心）

每一**类**错误，由专门的一层捕获；"完整验证"=矩阵被填满。

| 错误类别 | 捕获层 | 手段 | 状态 |
|---|---|---|---|
| MPC 数值解算正确性 | 单元 | `verify_mpc_step.py`（acados 解 vs 解析）、`verify_mpc_vs_pid.py` | ✅ 已验 |
| QP 后端等价性 | 单元 | ADMM(qp_admm.m) vs acados HPIPM 同解一致性 | ✅ 行为一致 |
| **算法逻辑**（编队/碰撞/邻居预测/参考前馈/速度合成） | **算法交叉验证** | **Simulink 移植件**（独立栈，12 场景，见 §2/§3） | ✅ 12/12（零 FAIL） |
| 控制律稳定性（收敛、不震荡、端点行为） | 算法交叉验证 + 集成 | Simulink + SITL（Tier1/2 治塌缩/漂移） | ✅ 两侧已验 |
| 安全保护层（围栏/碰撞地板/估计门/失效状态机） | 单元 + 注入 | `safety_filter.py` 自测 10/10；Simulink 注入故障验证触发 | 自测 ✅ / 注入待做 |
| EKF/出生校准/机间坐标系一致 | **集成（仅 SITL）** | PX4-Gazebo SITL（真 EKF），cross5/grid9 | ◐ 部分（S4/S5/S6 ✅；S7 ✅ 1 fallback；S23/S25/S26 待做） |
| 通信延迟/丢包降级路径 | 故障注入 | SITL S23/S25/S26（comms_delay/dropout/kill） | ☐ 待 Ubuntu 实跑 |
| 实时性（solve < 15ms on RPi4-ARM） | HIL/台架 | RPi 源码编 acados + `verify_mpc_step` benchmark | ☐ 待做 |
| 物理保真/真机闭环 | 渐进放飞 | solo1→pair2→trio3 低空增量放飞 | ☐ 待做 |

> 结论：Simulink 移植件**精准覆盖"算法逻辑"层**；它**无法**也**不应**覆盖 EKF/通信/物理层——那是 SITL 与真机的职责。把 Simulink 做成"PX4-Gazebo 完整迁移"，只会用更低保真度重复"集成层"，不新增任何错误类。

| trio3 hover | 0.000m / PASS | PASS（既往实测） | ✅ 定性一致 |
| solo1/pair2/trio3 × line | 0.56m / REVIEW | 〔待 SITL 定量补齐〕 | 同判定，定量待跑 |

## 4. 边界声明（诚实划界 = 严谨性，不是减分）

Simulink 移植件**刻意不含**（由 SITL/真机负责）：
- PX4 固件（EKF2 / 控制分配 / mixer / commander）—— 喂控制器的是世界 NED **真值**，无状态估计误差；
- Gazebo 物理（气动 / 旋翼推力 / 传感器噪声 / 风）；
- ROS2/DDS 通信层（真实延迟/丢包/LQ-3 链路）；
- world_birth 校准 / ref_alt 温漂 / EKF reset 等**真机会炸机**的失效模式；
- companion 安全保护层 `safety_filter`（独立验证）。

正因剥离这些工程噪声，移植件才能**单独、干净地验证算法逻辑**（form_err≈0 即其确定性体现）。

## 5. 结论与验证完成路径

1. **代码准确性已由 §1 矩阵中"单元 + 算法交叉验证"两层证明**（MPC 数值、算法逻辑 12/12 零 FAIL、ROS↔Simulink 定量交叉印证）。
2. **集成保真验证由 PX4-Gazebo SITL 承担**，非 Simulink；待完成项：
   - SITL 跑 **S23/S25/S26**（故障注入）验证通信降级路径；
   - S7 grid9 line 有 1 次 fallback 待确认（Simulink 侧 grid9 验证待补）；
   - RPi 台架 acados benchmark（实时性）→ 渐进放飞。
3. **"PX4-Gazebo 完整迁移进 Simulink"不在路径上**：它降低保真度、摧毁交叉验证独立性、且不解决炸机相关的任何失效模式——其能产出的"算法逻辑正确"结论，§2/§3 已给出。
